Magento 2 is one of the most infrastructure-demanding open-source platforms in existence. It doesn't run on a LAMP stack — it requires a specific PHP version with roughly 15 extensions, a dedicated search engine (MySQL search was dropped in 2.4.x entirely), in-memory session management to survive PHP-FPM restarts, and a full-page cache layer to be remotely usable under real traffic. A manual setup takes 4–6 hours and produces a result that's almost impossible to reproduce accurately.

Magento_DeployKit is my answer to that problem: seven ordered shell scripts that take a fresh Debian 12 VPS to a fully operational, production-hardened Magento 2.4.8 stack with a single sequential run. This post documents the engineering decisions behind it, the problems I hit, and the measurable results.

"A manual install takes 4–6 hours and produces a result that's hard to reproduce. This project reduces that to a single sequential run with predictable outcomes."

The Engineering Problem

Before writing a single line of shell script, I scoped the full dependency chain Magento 2.4.x actually requires — not what the quickstart guide suggests:

  • PHP 8.2-FPM with ~15 specific extensions (bcmath, gd, intl, soap, xsl, zip, and more)
  • Composer 2.x with authenticated access to the Magento Marketplace private repository
  • MySQL 8.0 with a correctly privileged app user, a separate read-only user, and log_bin_trust_function_creators=1
  • Elasticsearch 7.x — not 8.x, which Magento 2.4.x is not compatible with — with JVM heap tuning and OS-level file descriptor limits
  • Redis for both session storage and object cache — two separate Redis databases
  • Varnish 7 as the full-page cache layer fronting Nginx on port 80
  • Nginx handling both SSL termination (port 443) and as Varnish's backend (port 8080)
  • 2 GB swap to absorb Composer and DI compilation memory spikes on a 4 GB server

Every service has its own installation quirks, configuration surface, and failure mode. Automating all of it idempotently — so scripts can be re-run safely — required understanding each service at the level of what breaks when it's slightly wrong, not just how to install it when everything goes right.

Architecture & Design Decisions

The request path is the foundation of the whole topology. Mapping it explicitly before writing config was what made the port layout make sense:

# HTTP request path (port 80)
Browser
  │
  └─[HTTP :80]──→ Varnish (full-page cache)
                    └─[cache miss]──→ Nginx :8080 ──→ PHP-FPM ──→ Magento 2

# HTTPS request path (port 443)
Browser
  │
  └─[HTTPS :443]─→ Nginx (SSL termination) ──→ PHP-FPM ──→ Magento 2
                                               ├──→ MySQL 8         (persistent data)
                                               ├──→ Elasticsearch 7 (catalog search)
                                               ├──→ Redis           (sessions + object cache)
                                               └──→ Filesystem      (static assets)

Each key decision in this topology has a concrete reason:

Decision Rationale
Varnish :80, Nginx :8080 Varnish cannot handle TLS. NGINX terminates SSL on 443 independently. Port 8080 is the backend Varnish proxies to on cache miss — clean separation of concerns, no port conflicts.
Dedicated PHP-FPM pool Running a named pool under the web user's UID means socket permissions align without chmod workarounds. Process limits are scoped to Magento, not shared with other apps.
Elasticsearch over MySQL search Magento 2.4+ dropped built-in MySQL catalog search. ES 7.x is the minimum supported version and provides proper tokenisation, relevance scoring, and horizontal scalability.
Redis for sessions PHP file sessions don't survive PHP-FPM process recycling cleanly. Redis provides atomic session writes and persists across restarts — no cart loss under normal operations.
2 GB swap on 4 GB RAM Composer's dependency resolution and setup:di:compile are memory spikes, not sustained usage. Swap prevents OOM kills during install without requiring a larger droplet.
Dedicated MySQL users (app + read-only) Principle of least privilege. The app user has only the permissions Magento needs. A separate read-only user is safe to expose to phpMyAdmin and external tooling without access risk.

The 7-Script Automation System

The system is structured as seven ordered, idempotent shell scripts. Running them out of order will fail — by design. Each script has a defined responsibility and hands off a clean state to the next.

01
01-System-Prep.sh
Creates a 2 GB swap file (critical for Composer and DI compilation on constrained RAM), sets nproc 4096 process limits for the web user, imports GPG keys for MySQL and Elasticsearch apt repositories, and installs base utilities.
02
02-PHP-MySQL-Installation.sh
Installs PHP 8.2-FPM with all Magento-required extensions. Installs Composer with checksum validation. Sets up MySQL 8.0, creates the Magento database, an app-level DB user, and a read-only DB user for safer external access.
03
03-NGINX-Composer-Setup.sh
Installs NGINX configured to run under the web user. Sets up auth.json with Magento Marketplace credentials so composer install can pull proprietary packages without interactive prompts.
04
04-Elasticsearch-Redis-Installation.sh
Installs Elasticsearch 7.x from the official Elastic apt repository. Configures JVM heap (2 GB), vm.max_map_count=262144, and file descriptor limits — all required for ES to start cleanly on a constrained server. Installs Redis for sessions and object cache.
05
05-Magento-Installation.sh
Runs bin/magento setup:install with full DB, search, cache, and admin configuration. Deploys static content, reindexes all indexers, and sets correct permissions on var/, pub/, and app/etc/.
06
06-Web-Server-Config.sh
Creates a dedicated PHP-FPM pool with a named socket. Writes NGINX site configs for Magento (port 8080) and phpMyAdmin. Generates a self-signed SSL certificate. Installs and configures Varnish to front NGINX on port 80.
07
07-Final-Config-Checks.sh
Sets final file ownership (2755/644), creates a read-only Linux user with ACL access for safe file inspection, persists log_bin_trust_function_creators=1 in MySQL config, configures Magento FPC via Varnish, and prints a full health-check summary across all services.

Key shell patterns used throughout

The scripts aren't just sequences of apt install calls. Several patterns make them production-appropriate:

bash# Idempotency: check before creating
if [ ! -f /swapfile ]; then
  fallocate -l 2G /swapfile
  chmod 600 /swapfile
  mkswap /swapfile
  swapon /swapfile
  echo '/swapfile none swap sw 0 0' >> /etc/fstab
fi

# Checksum validation before installing Composer
php -r "if (hash_file('sha384', 'composer-setup.php') === 'EXPECTED_HASH') {
  echo 'Installer verified'; } else { echo 'Installer corrupt'; unlink('composer-setup.php'); }"

# Scoped process limits for the web user
echo "test-ssh  soft  nproc  4096" >> /etc/security/limits.conf
echo "test-ssh  hard  nproc  4096" >> /etc/security/limits.conf

Performance Results

The Varnish full-page cache impact is the most direct validation that the multi-layer architecture is working correctly. Measured on a catalog search result page using browser DevTools:

Cold cache — full stack round trip
3.0 MB
Data transferred — PHP, MySQL, Elasticsearch all executed — 1.60s to finish
Warm cache — served from Varnish memory
77.4 kB
~97% less data transferred — 910ms to finish. PHP, MySQL, ES never touched.

The 97% data reduction isn't a tuning win — it's proof the architecture is correctly configured. The Varnish VCL is correctly hashing cache keys, PHP-FPM is correctly sending X-Magento-Cache-Control headers, and the Nginx backend is correctly proxying cache-miss requests. All three services talking to each other correctly is the real measurement here.

What I Learned — 6 Hard Lessons

These aren't retrospective observations — each one is something that caused a real failure during the build and forced a deeper understanding of the underlying system.

1. Varnish and TLS fundamentally cannot coexist on the same port

Varnish has no TLS support — it proxies plain HTTP only. The solution required a deliberate two-frontend design: NGINX terminates SSL on 443, Varnish owns port 80, and NGINX listens on 8080 as Varnish's backend. Getting this topology right required tracing the full request path on paper before writing a single config line. Starting from configs and hoping they'd work together would have taken days.

2. PHP-FPM socket permissions aren't automatic — and silent 502s are the symptom

A generic www pool running under a different UID than NGINX's worker process causes 502 Bad Gateway errors with no obvious log entry on the PHP side. The fix — a dedicated named pool under the same user as the web root — taught me how FastCGI socket ownership actually works at the kernel level, not just the symptom level.

3. Elasticsearch 7 on 4 GB RAM needs explicit OS-level preparation

Elasticsearch silently refuses to start if vm.max_map_count is below 262144. It also requires at least 65535 open file descriptors per process. Neither limit is set correctly on a default Debian install. Getting ES to start reliably required adding these to /etc/sysctl.conf and /etc/security/limits.conf before the service is ever started — settings that aren't in the Magento install docs at all.

4. Swap matters more than RAM for install-time stability

Composer's dependency resolution and setup:di:compile both spike memory briefly — they're not sustained workloads. Adding a 2 GB swap file prevented two separate OOM kills during the initial install without requiring a larger (more expensive) droplet. The lesson: understand the shape of memory usage, not just the peak.

5. git rm --cached is the real .gitignore fix

Adding a file to .gitignore does nothing if the file is already tracked in the index. Discovering this after committing .DS_Store and a config file containing credentials made the distinction between Git's index and the working tree concrete — something that reading about it never quite did.

6. Redis session writes protect against a real failure mode

File-based PHP sessions can be wiped when PHP-FPM is restarted or the process tree is recycled under load. Redis-backed sessions are atomic and process-independent. I only fully understood why this mattered after tracing a cart-loss scenario: a user's session was in a file that PHP-FPM had an exclusive lock on during a restart. The file wasn't corrupted — it just wasn't accessible for the duration. Redis removes that entire class of failure.

Future Roadmap

The current state is a solid production baseline, but several things would make it genuinely enterprise-grade:

  • Let's Encrypt via Certbot — replace the self-signed certificate with automated renewal. The topology is already correct for this; it's purely a configuration change.
  • Export Magento's VCL from the admin panel — Magento can generate a Varnish VCL file that's fully aware of its cache control logic. Applying this instead of a generic VCL would make cache hit rates significantly higher.
  • Centralise all variables into 00-vars.sh — currently each script has variables at the top. A single sourced config file would be the right abstraction for team use.
  • CI/CD pipeline with zero-downtime deploys — using a blue-green deployment pattern with Nginx upstream switching for Magento upgrades without downtime.
  • Secrets management — HashiCorp Vault or Ansible Vault to eliminate hardcoded credentials entirely, even from the 00-vars.sh approach.
